**총평:** 시스템은 단일 개발자가 빠르게 프로토타이핑한 구조를 그대로 운영에 이관한 형태로, 기능적으로는 동작하나 유지보수성·보안·운영 안정성 측면에서 중대한 개선이 필요하다.

## 2. 시스템 개요

agri-news-brief는 GitHub Actions 기반의 농산물(원예) 뉴스 브리핑 자동화 시스템이다. 매 영업일 KST 06:00에 실행되어 Naver News API로 기사를 수집하고, OpenAI API로 요약을 생성한 뒤, HTML 브리핑 페이지를 GitHub Pages에 게시하고, 카카오톡으로 알림을 전송한다.

**구성 모듈 (11개 Python 파일)**

| 모듈 | 줄 수 | 비중 | 역할 |
|------|-------|------|------|
| main.py | 25,196 | 92.5% | 전체 파이프라인 (수집·필터·점수·요약·렌더링·배포·카카오) |
| hf_semantics.py | 456 | 1.7% | Hugging Face 시맨틱 리랭킹 |
| ux_patch.py | 460 | 1.7% | 아카이브 UX 패치 |
| replay.py | 308 | 1.1% | 스냅샷 직렬화/역직렬화 |
| collector.py | 241 | 0.9% | Naver API 호출 |
| io_github.py | 173 | 0.6% | GitHub API 래퍼 |
| schemas.py | 158 | 0.6% | 데이터 타입 정의 |
| observability.py | 68 | 0.3% | 메트릭 카운터 |
| orchestrator.py | 61 | 0.2% | 실행 흐름 분기 |
| retry_utils.py | 54 | 0.2% | 재시도 유틸리티 |
| ranking.py | 22 | 0.1% | 기사 정렬 키 |

## 3. 감사 영역별 상세 결과

### 3.1 아키텍처 및 코드 품질

#### 지적 3.1.1 [심각] main.py 모놀리스 구조

main.py가 전체 코드베이스의 92.5%(25,196줄)를 차지하며, 485개 함수와 3개 클래스가 단일 파일에 집중되어 있다. 이는 유지보수, 테스트, 코드 리뷰를 사실상 불가능하게 만든다.

**근거:**
- 총 27,196줄 중 25,196줄이 main.py에 존재
- 함수 485개 중 7개가 500줄 이상 (아래 표 참조)
- main.py가 나머지 10개 모듈 전부를 import (높은 결합도)

| 함수명 | 시작 줄 | 줄 수 | 위험도 |
|--------|---------|-------|-------|
| select_top_articles | 12,959 | 2,351 | 심각 |
| render_managed_commodity_board_html | 20,783 | 1,958 | 심각 |
| render_index_page | 22,741 | 793 | 높음 |
| is_relevant | 10,748 | 699 | 높음 |
| build_sections_from_raw | 18,072 | 638 | 높음 |
| collect_candidates_for_section | 16,536 | 567 | 높음 |
| compute_rank_score | 11,447 | 564 | 높음 |

select_top_articles 함수는 2,351줄로, 기사 필터링·점수화·중복 제거·시맨틱 조정·소스 다양성 제한을 모두 단일 함수에서 수행한다. 이 함수 하나의 결함이 전체 브리핑 품질을 좌우하며, 단위 테스트가 사실상 불가능하다.

#### 지적 3.1.2 [심각] 통제되지 않는 전역 상태

main.py에 약 602개의 변경 가능한(mutable) 전역 변수가 존재하며, 11곳에서 명시적 `global` 선언을 통해 함수 내에서 전역 상태를 변경한다.

**대표 사례:**
- `RECENT_HISTORY_CANON: set[str]` (줄 474) — 크로스데이 중복 방지용 캐시
- `_COND_PAGING_EXTRA_CALLS_USED` (줄 602) — 조건부 페이징 예산 추적
- `DEBUG_DATA: JsonDict` (줄 687) — 디버그 리포트 상태
- `_LAST_COMMODITY_BOARD_SOURCE_BY_SECTION` (줄 2,481) — 품목보드 소스 캐시

**영향:** 함수가 비결정적(non-deterministic)으로 동작하여 테스트 시 복잡한 전역 상태 설정/해제가 필요하고, 병렬 실행 시 경합 조건(race condition)이 발생할 수 있다.

#### 지적 3.1.3 [높음] 하드코딩된 설정 데이터 (1,700줄 이상)

품목 카탈로그, 섹션 정의, 검색 쿼리 뱅크, 토픽 패턴 등이 Python 소스코드 내에 직접 정의되어 있다.

- MANAGED_COMMODITY_GROUP_SPECS: 342줄 (줄 1,502~1,844)
- MANAGED_GROUP_RECALL_QUERY_BANK: 386줄 (줄 1,845~2,231)
- SECTIONS 정의: 484줄 (줄 2,231~2,715)
- 토픽 패턴/컨텍스트 정규식: 1,000줄 이상 (줄 3,263~)

**영향:** 품목 추가·쿼리 변경 시 반드시 코드 수정과 배포가 필요하며, 운영자가 개발자 없이는 설정을 변경할 수 없다.

#### 지적 3.1.4 [중간] 타입 안전성 우회

mypy strict 모드가 보조 모듈 8개에만 적용되며, 가장 핵심인 main.py와 replay.py는 타입 검사에서 제외되어 있다.

- mypy.ini에 `files = collector.py, io_github.py, retry_utils.py, schemas.py, ux_patch.py, ranking.py, orchestrator.py, observability.py`로 명시 (main.py 미포함)
- 전체 타입 어노테이션 커버리지 92% (양호하나, 핵심 모듈이 검증에서 빠짐)

#### 지적 3.1.5 [중간] 독스트링 커버리지 부족

485개 함수 중 170개(35%)만 독스트링을 보유하고 있어, 315개 함수는 코드를 직접 읽어야만 동작을 이해할 수 있다.

### 3.2 보안

#### 지적 3.2.1 [심각] 로컬 환경 파일에 실제 API 키 노출

.env.local, .env.dev.local, .env.final.local 파일에 운영 중인 실제 API 키가 평문으로 저장되어 있다. .gitignore에 올바르게 제외 규칙이 있으나, 작업 디렉토리에 파일이 존재하므로 레포지토리 접근 권한이 탈취될 경우 전체 크리덴셜이 유출된다.

**노출된 크리덴셜 목록:**
- OpenAI API Key (`sk-proj-YSb3uESU...`)
- GitHub Personal Access Token (`github_pat_11AVEWOCQ...`)
- Kakao Refresh Token
- HuggingFace Token (`hf_KrzGJu...`)
- Naver Client ID/Secret

**즉시 조치 필요:** 모든 노출된 키를 즉시 로테이션(갱신)하고, git history에 커밋된 적이 없는지 확인해야 한다.

#### 지적 3.2.2 [높음] 로그 마스킹 패턴 불완전

`_SECRET_PATTERNS` (main.py 줄 210~215)에 정의된 마스킹 규칙이 다음 패턴을 포함하지 않는다:

- `sk-proj-*` (OpenAI 키 패턴)
- `github_pat_*` (GitHub 토큰 패턴)
- `hf_*` (HuggingFace 토큰 패턴)
- Naver Client ID/Secret 패턴
- `api_key=` 파라미터 패턴

에러 로그를 통해 민감 정보가 GitHub Actions 콘솔에 노출될 수 있다.

#### 지적 3.2.3 [높음] Kakao 토큰 응답 검증 미흡

main.py 줄 23,796에서 Kakao OAuth 토큰 갱신 응답을 `j["access_token"]`으로 직접 접근하고 있으나, 응답에 해당 키가 없을 경우 KeyError가 발생하며 적절한 에러 메시지 없이 중단된다.

#### 지적 3.2.4 [중간] 토큰 로테이션 전략 부재

모든 API 키와 토큰이 무기한으로 사용되며, 정기적 로테이션 절차나 만료 알림이 없다. GitHub Personal Access Token은 fine-grained token보다 넓은 권한 범위를 가질 수 있어 추가적인 위험이 존재한다.

### 3.3 에러 처리 및 복원력

#### 지적 3.3.1 [높음] 예외의 무시적 처리 (Silent Swallowing) — 56건

main.py의 227개 try/except 블록 중 56건(24.8%)이 `except Exception: pass`로 예외를 무시하고 있다. 이는 잠재적 버그를 숨기고 디버깅을 극도로 어렵게 만든다.

**주요 발생 위치:** 줄 222, 301, 305, 324, 713, 739, 750, 760, 775, 806, 811, 813, 841, 2619, 3164, 3252, 10191, 10426, 11991, 12348 외 36건

**분류:**
| 유형 | 건수 | 비율 |
|------|------|------|
| 무시(pass only) | 56 | 24.8% |
| 폴백 값 할당 | 70 | 31.0% |
| 로그 후 계속 | 41 | 18.1% |
| 값 반환 | 32 | 14.2% |
| 기타 | 27 | 11.9% |
| **재발생(re-raise)** | **0** | **0.0%** |

특히 예외를 재발생(re-raise)시키는 케이스가 0건이라는 점은, 모든 오류가 내부에서 흡수되어 상위 호출자에게 전파되지 않음을 의미한다.

#### 지적 3.3.2 [중간] 서킷 브레이커 패턴 미구현

Naver API와 Kakao API에 대해 비재시도(non-retryable) 에러 분류는 구현되어 있으나, OpenAI API에는 인증 실패와 일시적 오류를 구분하지 않아 인증 문제 시에도 최대 재시도 횟수만큼 불필요한 호출을 반복한다.

**긍정적 사항:** 모든 외부 API에 대해 fail-open(실패 시 워크플로우 중단하지 않음) 패턴이 적용되어 있으며, 재시도 횟수가 제한되어 있어 무한 루프 위험은 없다.

| API | 최대 재시도 | 백오프 기본값 | 백오프 상한 | 제한됨 |
|-----|-----------|-------------|-----------|-------|
| Naver | 6회 | 1.0초 | 20.0초 | 예 |
| OpenAI | 3회 | 0.8초 | 20.0초 | 예 |
| Kakao | 6회 | 0.8초 | 15.0초 | 예 |
| HTTP (urllib3) | 3회 | 0.3초 | — | 예 |

### 3.4 테스트 및 CI/CD

#### 지적 3.4.1 [높음] CI 파이프라인 강제성 미흡

.github/workflows/ci.yml에서 구문 검사, mypy, 단위 테스트 3개 단계가 모두 `continue-on-error: true`로 설정되어 있다. 최종 단계에서 실패를 검출하는 로직이 있으나, branch protection rule이 별도로 설정되어 있지 않으면 실패한 PR이 머지될 수 있다.

#### 지적 3.4.2 [중간] 테스트 구성 불균형

23개 테스트 파일, 399개 테스트 함수가 존재하나, 대부분이 통합 테스트이며 단위 테스트 비중이 매우 낮다.

| 유형 | 파일 수 | 비율 |
|------|---------|------|
| 통합 테스트 | 21 | 91% |
| 단위 테스트 | 2 | 9% |

- observability.py(메트릭 인프라)에 대한 테스트가 존재하지 않음
- 린팅 도구(flake8/pylint) 미적용
- 코드 커버리지 리포팅 미구현

#### 지적 3.4.3 [낮음] main.py의 mypy 검사 미적용

전체 코드베이스의 92.5%를 차지하는 main.py가 mypy strict 모드에서 제외되어 있어, 타입 관련 결함이 런타임에서만 발견될 수 있다.

### 3.5 운영 안정성

#### 지적 3.5.1 [높음] 외부 알림(Alerting) 미구현

워크플로우 실패, API 오류 누적, 연속 실패 등에 대한 외부 알림 채널(Slack, PagerDuty 등)이 연동되어 있지 않다. 운영자는 GitHub Actions 콘솔을 수동으로 확인해야만 장애를 인지할 수 있다.

- 메트릭은 프로세스 내 메모리에만 존재하며, 워크플로우 종료 시 소실됨
- Admin 대시보드의 health.json은 항상 `"status": "ok"`을 반환 (실제 장애 감지 로직 없음)
- "24시간 내 성공적 실행이 없음"과 같은 SLA 모니터링 부재

#### 지적 3.5.2 [높음] workflow_dispatch 입력값 검증 부재

rebuild.yml, maintenance.yml 등의 수동 실행 워크플로우에서 입력값(report_date, backfill_days 등)에 대한 형식 검증이 없다. 잘못된 날짜 형식(예: "2026-13-01")이 입력되면 45~120분 실행 후 Python 코드 내부에서 실패한다.

#### 지적 3.5.3 [중간] 크로스데이 중복 방지 경합 조건

daily.yml의 concurrency 설정이 `cancel-in-progress: true`이므로, 수동 재실행과 자동 실행이 겹칠 경우 이전 실행이 취소되면서 상태 파일(.agri_state.json)의 일관성이 깨질 수 있다.

#### 지적 3.5.4 [중간] GitHub API 쓰기 충돌 복구 제한

io_github.py의 SHA 기반 낙관적 잠금(optimistic locking)에서 충돌 시 SHA 재조회를 1회만 시도한다. 3개 이상의 동시 프로세스가 같은 파일을 수정할 경우 쓰기가 실패할 수 있다.

### 3.6 데이터 무결성

#### 지적 3.6.1 [중간] 상태 파일 손상 시 무경고 열화

.agri_state.json이 손상(JSON 파싱 실패)되면 `{"last_end_iso": None}`을 반환하며, 크로스데이 중복 방지가 자동 비활성화된다. 경고 로그 없이 이전 기사가 재노출될 수 있다.

#### 지적 3.6.2 [중간] 매니페스트 복원 취약점

아카이브 매니페스트 구축 시 GitHub API의 디렉토리 목록 조회가 실패하면, 매니페스트가 현재 날짜 1건으로 리셋된다. 이 경우 네비게이션 링크가 일시적으로 깨질 수 있다.

#### 지적 3.6.3 [낮음] 스냅샷 스키마 마이그레이션 경로 부재

replay.py의 SNAPSHOT_SCHEMA_VERSION이 1로 고정되어 있으며, 스키마 변경 시의 마이그레이션 전략이 문서화되어 있지 않다. 향후 필드 추가 시 기존 스냅샷과의 호환성 문제가 발생할 수 있다.

## 5. 시스템 고도화 방안

### Phase 1: 긴급 조치 (1~2주)

**5.1.1 크리덴셜 전수 교체 및 보안 강화**

모든 노출된 API 키를 즉시 로테이션한다. git history에 커밋된 적이 없는지 `git log --full-history -- .env.local`로 확인하고, 만약 커밋된 적이 있다면 해당 커밋을 제거하거나 레포를 재생성한다. 추가로 pre-commit hook에 `detect-secrets` 또는 `git-secrets`를 통합하여 향후 크리덴셜 커밋을 원천 차단한다.

**5.1.2 CI/CD 강제성 확보**

ci.yml의 `continue-on-error: true`를 제거하고, GitHub branch protection rule에서 CI 통과를 PR 머지 요건으로 설정한다.

**5.1.3 workflow_dispatch 입력 사전 검증**

모든 수동 실행 워크플로우에 날짜 형식 검증(YYYY-MM-DD regex), 숫자 범위 검증을 첫 번째 step으로 추가하여, 잘못된 입력이 45분 이상 실행된 후 실패하는 상황을 방지한다.

---

### Phase 2: 구조 개선 (1~3개월)

**5.2.1 main.py 모듈 분리**

main.py를 다음 도메인 모듈로 분리한다:

| 신규 모듈 | 대상 함수군 | 예상 규모 |
|----------|-----------|----------|
| `config.py` | 설정 로딩, 환경변수, 상수 정의 | ~800줄 |
| `article_selection.py` | select_top_articles, is_relevant, compute_rank_score | ~4,000줄 |
| `normalization.py` | URL/텍스트 정규화, 중복 키 생성 | ~500줄 |
| `validation.py` | 106개 is_* 검증 함수 | ~2,000줄 |
| `summarizer.py` | OpenAI 요약 호출, 캐시 관리 | ~500줄 |
| `rendering.py` | HTML 렌더링 (render_daily_page, render_index_page 등) | ~5,000줄 |
| `commodity_board.py` | 품목보드 관련 로직 | ~3,000줄 |
| `kakao.py` | 카카오톡 메시지 구성/전송 | ~500줄 |
| `state.py` | 상태 파일 로드/저장, 크로스데이 중복 방지 | ~500줄 |

**5.2.2 설정 외부화**

1,700줄 이상의 하드코딩된 품목 카탈로그, 섹션 정의, 쿼리 뱅크를 JSON/YAML 파일로 분리하고, Pydantic 기반 검증 스키마를 도입한다. 이를 통해 운영자가 코드 수정 없이 품목·쿼리를 관리할 수 있게 한다.

**5.2.3 전역 상태 제거 및 의존성 주입**

전역 변수를 `PipelineContext` 또는 `RunConfig` 같은 컨텍스트 객체로 캡슐화하고, 함수 파라미터로 전달(의존성 주입)하는 방식으로 전환한다. 이를 통해 함수의 결정성(determinism)을 확보하고 단위 테스트를 용이하게 한다.

**5.2.4 로그 마스킹 보강**

`_SECRET_PATTERNS`에 다음 패턴을 추가한다:
- `sk-proj-[A-Za-z0-9_-]+` (OpenAI)
- `github_pat_[A-Za-z0-9_]+` (GitHub)
- `hf_[A-Za-z0-9]+` (HuggingFace)
- `X-Naver-Client-Secret: .+` (Naver)
- `api_key=[^&\s]+` (일반)

### Phase 3: 운영 고도화 (3~6개월)

**5.3.1 모니터링 및 알림 체계 구축**

- GitHub Actions 실패 시 Slack webhook 알림 발송
- "24시간 내 성공적 실행 없음" 감지 및 경고
- API별 에러율 임계값 초과 시 알림 (예: OpenAI 실패율 > 30%)
- 데이터 품질 검증 자동화: 섹션별 기사 수 < 2 또는 > 20이면 경고

**5.3.2 테스트 전략 재설계**

- 단위 테스트 비중을 50% 이상으로 확대 (현재 9%)
- 외부 API를 mock으로 대체하여 테스트 속도와 안정성 확보
- 코드 커버리지 측정 도입 (coverage.py) 및 CI 리포팅
- observability.py, 신규 분리 모듈에 대한 테스트 추가
- 린팅 도구(flake8 또는 ruff) CI 파이프라인에 추가

**5.3.3 상태 관리 견고화**

- .agri_state.json에 체크섬 필드를 추가하여 손상 감지
- 이전 2개 버전의 상태 파일을 백업으로 유지 (state.json.bak.1, .bak.2)
- JSON 파싱 실패 시 이전 버전에서 자동 복구
- 상태 파일에 버전 필드를 추가하여 향후 호환성 보장

**5.3.4 서킷 브레이커 도입**

OpenAI API에 대해 인증 실패(401/403)와 일시적 오류(429/5xx)를 구분하는 로직을 추가하고, 인증 실패 시 즉시 중단하여 불필요한 재시도를 방지한다.

---

### Phase 4: 장기 고도화 (6~12개월)

**5.4.1 마이크로서비스 고려**

파이프라인 단계를 독립 실행 가능한 단위로 분리하여, 수집(collector)→필터링(selector)→요약(summarizer)→렌더링(renderer)→배포(publisher) 각 단계를 개별 워크플로우 job으로 구성한다. 각 단계 간 데이터를 artifact로 전달하면, 특정 단계만 재실행하는 것이 용이해진다.

**5.4.2 설정 관리 UI**

품목 카탈로그, 섹션 정의, 쿼리 뱅크를 관리할 수 있는 경량 웹 관리 화면을 구축한다. GitHub API를 통해 설정 파일을 직접 편집하는 방식으로, 별도 서버 없이도 GitHub Pages 기반으로 구현할 수 있다.

**5.4.3 관측성(Observability) 대시보드 고도화**

현재 "항상 OK"만 표시하는 admin 대시보드를 실제 파이프라인 건강 지표로 확장한다: 수집 성공률, 요약 성공률, 섹션별 기사 수, API 응답 시간, 재시도 횟수, 캐시 히트율 등을 시계열로 추적한다.

**5.4.4 시크릿 관리 체계화**

HashiCorp Vault 또는 GitHub Environment Secrets를 활용한 시크릿 관리 체계를 도입하고, 분기별 토큰 로테이션 절차를 수립한다. 토큰 만료 7일 전 자동 알림을 구현한다.
